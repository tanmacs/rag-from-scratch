# Reflection

This pipeline has five explicit components because the assignment is about understanding the mechanics of retrieval-augmented generation rather than hiding them behind a framework. The document loader matters because the quality and scope of the knowledge base determine what the system can possibly answer. If the source documents are too short, too noisy, or inconsistent in style, retrieval quality drops immediately. The chunker matters because retrieval operates on chunks rather than whole files. A chunk that is too small loses context, while a chunk that is too large mixes unrelated facts and makes ranking less precise. I used fixed-size overlapping character chunks so the implementation stays simple and transparent, and I kept the overlap at fifty characters so facts near boundaries are less likely to be cut apart.

The embedder converts chunks and queries into vectors in a shared semantic space. That step is important because it lets the system match related wording even when the query does not repeat the document text exactly. The retriever then computes cosine similarity with NumPy only, which makes the retrieval logic easy to inspect. Finally, the generator takes the retrieved chunks and the user question and turns them into a natural-language answer. I added a strict instruction telling the model to answer only from context and to say it does not know when the context is insufficient. That prompt design is central to grounding; without it, the model is far more likely to invent details.

The pipeline still fails in predictable ways. Character chunking is simple, but it can split sentences awkwardly and sometimes retrieves a fragment that contains the right keywords without enough surrounding explanation. The evaluation questions that ask for explanation or comparison can also expose retrieval weakness when two adjacent chunks from the same document are both needed. The generator is another weak point. A small local model such as FLAN-T5-small is convenient and lightweight, but it sometimes compresses or paraphrases too aggressively. That can make otherwise grounded answers feel incomplete or slightly imprecise.

If I had more time, the first change I would make is improving chunking and retrieval diagnostics. I would preserve sentence boundaries, log chunk-level scores during evaluation, and experiment with chunk size and overlap to see how Precision@3 changes. After that, I would add a reranking step or better prompt formatting while still staying within the assignment constraints. One limitation of cosine similarity is that it measures directional similarity between vectors but ignores document structure, discourse importance, and exact factual dependency. Two chunks can have similar embeddings even when only one contains the precise fact needed for a question, so semantic closeness alone is not always enough for reliable retrieval.
